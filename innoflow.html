Overview

Universities and training providers are faced with the challenge of overseeing many, often highly technical, projects at a time. As it stands, student software engineering projects are largely opaque in that key updates and progression information rarely make it past the students directly involved. Individual students’ contributions are equally isolated across multiple projects, making it difficult for educational entities and employers to appraise students effectively. Assessment will typically take place at the end where probing a project for qualities that should have been observable from the start proves most difficult. This poses a problem for project management and assessment, where human resources are limited while pressure on students and staff to perform well is comparatively high.

To help address this disparity, we developed an software platform called InnoFlow where project development behaviour is made largely transparent to supervisors and where feedback and assessment are made top priorities. Previous work has largely focused on automating assessment of programming exercises and peer review. We extended this approach to more open-ended projects, integrating it with the peer review process to further reduce the burden on teaching staff. Moreover, projects are grouped into classes and their source-control repositories analysed to produce quantifiable metrics as a means of assessment and self-improvement.

Approach

InnoFlow was designed to solve the three listed problems:

  • Staff members may find it difficult or time-consuming to assess larger, open-ended projects.
  • Evidence of learning outcomes is scattered across commits, and can often be inaccessible to staff.
  • Students’ contributions are spread out across multiple projects, much to the detriment of students, staff and even employers.

Passive Assessment Summaries

InnoFlow firstly aims to implement automatic tracking of development activity, improving project transparency for supervisors and peers. This is achieved by progressively tracking commit, code review and peer-feedback actions triggered by the project team, thus allowing InnoFlow to compute these phenomena as metrics.

Active Innovation Showcasing

InnoFlow’s secondary goal is to provide users with the opportunity of highlighting and storing their innovative work, gathering together what would otherwise be siloed across different project repositories. While also benefiting the assessment process for teaching staff, individuals then have the ability to collect their most valuable work examples as part of their InnoFlow profiles, promoting them to potential employers.

Components

The InnoFlow system is portioned into four independently working components. It is important to note that we chose VSTS as the project management service for our architecture. The reason for this decision can be found as part of the project report.

InnoFlow API

Our API is firstly responsible for interacting with various VSTS APIs (Accounts and Proles, Git, Projects and Teams and Service hooks) to extract user and project related VSTS data. This communication is crucial for authorising InnoFlow users with VSTS, extracting users’ VSTS projects, establishing VSTS web hooks and retrieving commit related information. Secondly, the API provides all communication/data endpoints for the remaining InnoFlow components, thus representing the backbone of the entire InnoFlow system.

InnoFlow Web App

Our web app, a third-party VSTS extension (see project report for explanation), represents the main frontend visualisation and project management interface for the InnoFlow API. First of all, it allows students to enrol their VSTS projects into classes, view commits, start code review discussions and visualise metrics about themselves and their project. Additionally, it showcases students’ innovative code submissions. Teaching staff members can create and manage classes, as well as compare enrolled projects with reference to their metric scores.

InnoFlow IDE Extension

Our IDE extension allows users to highlight and comment on their code within the Visual Studio Code environment, before submitting via the InnoFlow API. Note that the IDE extension is available on the Visual Studio Marketplace.

InnoFlow Chrome Extension

Our Chrome extension provides non-InnoFlow users, especially employers, with a method of searching for students’ innovative code samples, demonstrating a simple use case for the InnoFlow API.

Contribution

  • System Architect: One of my first contribution was to construct InnoFlow’s system architecture. I designed the VSTS communication mechanisms that allow InnoFlow to extract VSTS data by using the various VSTS APIs, thus establishing the entire logistics for building key features such as authorising InnoFlow users with VSTS, extracting users’ VSTS projects, creating VSTS web hooks for monitoring version control activities or retrieving commit related information. Secondly, I designed the information flow links between our system components, i.e. the data communication between InnoFlow’s API and its remaining components.
  • Backend Developer: My most notable contribution was the development and testing of the InnoFlow API using the Laravel (v 5.3) framework, which interacts with various VSTS APIs to extract user and project related VSTS data but more importantly provides communication/data endpoints for each extension component of the system.
  • Database Developer: I was further responsible for managing InnoFlow’s database schema using MySQL as its DBMS. This comprises the implementation of database seeders for mocking users and innovations in order to demonstrate the API’s user search and innovation retrieval functionalities. The resulting schema both holds tables reflecting InnoFlow concepts such as users, classes, innovations, code reviews or comments but also VSTS related concepts as in projects, commits or accounts. The latter was crucial to avoid the overhead of retrieving the same VSTS data over and over again as well as to secure the InnoFlow system against VSTS outages. For the most part, the database schema is in 3rd normal form. However, some tables are only partially normalised in order to speed up complex database queries.
